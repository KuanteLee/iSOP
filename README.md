基於神經的多任務學習已經成功地應用於許多現實世界的大規模應用，如推薦系統。例如，在電影推薦中，除了向用戶提供他們傾向於購買和觀看的電影外，系統還可能對用戶事後喜歡的電影進行優化。通過多任務學習，我們旨在建立一個單一的模型，同時學習這些多個目標和任務。然而，常用的多任務模型的預測質量往往對任務之間的關係很敏感。因此，研究任務特定目標和任務間關係之間的建模交易很重要。 <br>
<br>
在這項工作中，我們提出了一種新的多任務學習方法-- Multi-gate Mixture-of-Experts（MMoE），它明確地從數據中學習任務關係模型。我們通過在所有任務中共享專家子模型，使 Mixture-of-Experts（MoE）結構適應多任務學習，同時也有一個g ating network 被訓練來優化每個任務。為了在具有不同程度的任務關聯性的數據上驗證我們的方法，我們首先將其應用於我們控制任務關聯性的合成數據集。我們表明，當任務相關性較低時，所提出的方法比基線方法表現得更好。我們還表明，根據訓練數據和模型初始化中不同程度的隨機性，MMoE結構帶來了額外的可訓練性好處。此外，我們證明了MMoE在實際任務中的性能改進，包括二進制分類基準和谷歌的大規模內容推薦系統。 <br>
<br>
近年來，深度神經網絡模型已成功應用於許多現實世界的大規模應用，如推薦系統[11]。這樣的推薦系統往往需要同時優化多個目標。例如，在推薦電影給用戶觀看時，我們可能希望用戶不僅購買和觀看電影，還希望他們事後喜歡這些電影，以便再來購買更多的電影。也就是說，我們可以建立模型來同時預測用戶的購買行為和他們的評價。事實上，許多大規模的推薦系統已經採用了多任務學習，使用深度神經網絡（DNN）模型[3]。<br>
<br>
研究人員報告說，多任務學習模型可以通過利用正則化和轉移學習來提高模型對所有任務的預測[8]。然而，在實踐中，多任務學習模型並不總是在所有任務上超過相應的單任務模型[23, 26]。事實上，許多基於DNN的多任務學習模型對諸如數據分佈差異和任務之間的關係等因素很敏感[15, 34]。來自任務差異的內在矛盾實際上會損害至少一些任務的預測，特別是當模型參數在所有任務之間廣泛共享時。<br>
<br>
以前的工作[4, 6, 8]通過假設每個任務的特定數據生成過程，根據假設測量任務差異，然後根據任務的差異程度提出建議，研究了多任務學習中的任務差異。然而，由於實際應用往往有更複雜的數據模式，測量任務差異和利用這些先前工作的建議方法往往是困難的。<br>
<br>
最近的一些工作提出了新的建模技術來處理多任務學習中的任務差異，而不依賴明確的任務差異測量[15, 27, 34]。然而，這些技術往往涉及到為每個任務添加更多的模型參數以適應任務差異。由於大規模的推薦系統可能包含數百萬或數十億的參數，這些額外的參數往往是受限制的，這可能會損害模型質量。由於服務資源有限，在實際生產環境中，這些參數的額外計算成本也往往令人望而卻步。<br>
<br>
在本文中，我們提出了一種基於新穎的 MMoE 結構的多任務學習方法，其靈感來自 MoE 模型[21]和最近的 MoE 層[16, 31]。 MMoE明 確地對任務關係進行建模，並學習任務特有的功能，以利用共享表徵。它允許自動分配參數以捕獲共享任務信息或任務特定信息，避免了為每個任務添加許多新參數的需要。<br>
<br>
MMoE的主幹是建立在最常用的共享底層多任務DNN結構上[8]。共享底層模型結構如圖1（a）所示，輸入層之後的幾個底層在所有任務中共享，然後每個任務在底層表示之上有一個單獨的 "tower "網絡。我們的模型，如圖1（c）所示，沒有所有任務共享一個底層網絡，而是有一組底層網絡，每個底層網絡被稱為專家。在我們的論文中，每個專家都是一個前饋網絡。然後，我們為每個任務引入一個門控網絡。門控網絡接受輸入的特徵並輸出softmax門，用不同的權重組裝專家，允許不同的任務以不同的方式利用專家。然後，組裝專家的結果被傳遞到特定任務的 tower networks 中。通過這種方式，不同任務的門控網絡可以學習不同的專家組合模式，從而掌握任務關係。<br>
<br>
為了了解MMoE是如何針對不同的任務關聯度學習其專家和任務門控網絡的，我們進行了一個合成實驗，我們可以通過它們的皮爾遜相關度來測量和控制任務關聯度。與[24]類似，我們使用兩個合成回歸任務，並使用正弦函數作為數據生成機制來引入非線性。在這種設置下，我們的方法優於基線方法，特別是當任務相關性較低時。在這組實驗中，我們還發現MMoE更容易訓練，並在多次運行中收斂到更好的損失。這與最近發現的調製和門控機制可以提高訓練非凸的深度神經網絡的可訓練性有關[10, 19]。<br>
<br>
我們進一步評估了MMoE在基準數據集--UCI Census-income數據集上的性能，該數據集採用多任務問題設置。我們與幾個最先進的多任務模型進行了比較，這些模型通過軟參數共享來模擬任務關係，並觀察到我們方法的改進。<br>
<br>
最後，我們在一個真正的大規模內容推薦系統上測試了MMoE，在向用戶推薦項目時，兩個分類任務同時被學習。我們用數以千億計的訓練實例來訓練MMoE模型，並將其與一個共享底層生產模型進行比較。我們觀察到諸如AUC等光學指標的明顯改善。此外，我們的MMoE模型在現場實驗中持續改善在線指標。<br>
<br>
本文的貢獻有三點。首先，我們提出了一個新穎的 MMoE 模型，明確地對任務關係進行建模。通過調製和門控網絡，我們的模型在建模共享信息和建模任務特定信息之間自動調整參數化。第二，我們在合成數據上進行了控制實驗。我們報告了任務關聯性如何影響多任務學習的訓練動態，以及MMoE如何提高模型的表現力和可訓練性。最後，我們在真實的基準數據和一個擁有上億用戶和項目的大規模生產性推薦系統上進行了實驗。我們的實驗驗證了我們提出的方法在真實世界環境中的有效性和有效性。<br>
<br>
多任務模型可以學習不同任務之間的共同點和差異。這樣做可以提高每個任務的效率和模型質量[4, 8, 30]。其中一個廣泛使用的多任務學習模型是由Caruana[8, 9]提出的，它有一個共享底層模型結構，其中底層隱藏層在不同的任務中共享。這種結構大大降低了過度的風險，但由於所有的任務都需要在共享底層上使用相同的參數集，因此會出現由任務差異引起的優化衝突。<br>
<br>
為了了解任務關聯性對模型質量的影響，先前的工作使用合成數據生成，並操作不同類型的任務關聯性，以評估多任務模型的有效性[4-6, 8]。<br>
<br>
最近的一些方法不是在不同的任務間共享隱藏層和相同的模型參數，而是在任務特定的參數上添加不同類型的約束[15, 27, 34]。例如，對於兩個任務，Duong等人[15]在兩組參數之間添加了L-2約束。 Cross-stitch網絡[27]為每個任務學習了一個獨特的任務特定的隱藏層嵌入組合。 Yang等人[34]使用張量因子化模型為每個任務生成隱層參數。與共享底層模型相比，這些方法有更多的任務專用參數，當任務差異導致更新共享參數時，可以實現更好的性能。然而，大量的任務專用參數需要更多的訓練數據，在大規模模型中可能並不有效。<br>
<br>
在本文中，我們應用了深度學習中的一些最新發現，如參數調製和集合方法來為多任務學習的任務關係建模。在DNN中，集合模型和子網絡的集合已經被證明能夠提高模型的性能[9, 20]。<br>
<br>
Eigen等人[16]和Shazeer等人[31]將專家混合模型變成基本構件（MoE層），並在DNN中堆疊。 MoE層在訓練時間和服務時間都根據該層的輸入來選擇子網（專家）。因此，這個模型不僅在建模方面更加強大，而且通過在門控網絡中引入稀疏性來降低計算成本。同樣，PathNet[17]是為處理不同任務的藝術性通用智能而設計的，它是一個具有多層和每層內多個子模塊的巨大神經網絡。在為一項任務進行訓練時，隨機選擇多條路徑，由不同的工作人員並行訓練。最佳路徑的參數是固定的，新的路徑被選擇用於訓練新的任務。我們從這些工作中得到啟發，使用一個子網（專家）的集合來實現轉移學習，同時節省計算。<br>
<br>
由於分佈式機器學習系統的發展[13]，許多大規模的現實世界的應用已經採用了基於DNN的多任務學習算法，並觀察到質量的大幅提高。在多語言機器翻譯任務中，通過共享模型參數，具有有限訓練數據的翻譯任務可以通過與具有大量訓練數據的任務聯合學習來改善[22]。對於建立推薦系統，多任務學習被認為有助於提供上下文感知的推薦[28, 35]。在[3]中，一個文本推薦任務通過共享特徵表示和較低層次的隱藏層得到改善。在[11]中，一個共享底層模型被用來學習視頻推薦的排名算法。與這些先前的工作類似，我們在一個真實世界的大規模推薦系統上評估我們的建模方法。我們證明了我們的方法確實是可擴展的，並且與其他最先進的建模方法相比具有有利的性能。<br>

## index 使用時機
在正常情況下, 資料表的欄位 index 可以增進擷取資料列的效率, 但是在一個資料表上多建立 index 並不一定可以得到好處. 原因是, 當確認一個具有 index 的 DML 操作時, 意含著 ORACLE 資料庫需去更新該 index 列, 所以當建立 index 的欄位越多, 資料庫的更新負荷也越大, 有時還會適得其反, 整體速度反而更慢. 因此建議使用 index 時間如下: <br>

* 當資料表某個欄位儲存的值範圍很大時 <br>
* 某個可能會出現大量空值的欄位 <br>
* 常用於 WHERE 子句或合併條件的欄位 <br>
* 預期 index 的資料量經常 <4% 的大資料量資料表 <br>

然而, 在下列情況下, 不建議使用 index: <br>

* 不經常用在條件式的欄位 <br>
* 需要經常更新的資料表 <br>
* 小資料量的資料表 <br>
* 預期 index 的資料量經常 >4% 的大資料量資料表 <br>

# 簡報技巧

恢復原狀型：
1. 掌握到底發生了什麼狀況（掌握現狀）
2. 視情況所需，將危害控制在最小限度（緊急處理）
3. 查明原因所在（分析原因）
4. 思考該如何修復（原因處理）
5. 研究對策，避免再度損壞（防止復發）

防杜潛在型：
1. 設定一個一旦發生會造成困擾的問題（設定問題）
2. 查明造成此為題的誘因（查明誘因）
3. 思考預防問題發生的對策（預防對策）
4. 思考問題發生時的對策（發生時的因應對策）

追求理想型：
1. 分析問題點的優缺點（資產盤點）
2. 選定追求的理想 ＝ 目標（選定理想）
3. 思考可達成理想的對策（行動計畫）
